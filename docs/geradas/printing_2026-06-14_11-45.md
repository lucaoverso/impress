# printing

## 1. Resumo

O módulo **printing** é responsável por gerenciar todo o fluxo de impressão de documentos no sistema, incluindo o envio de arquivos para impressão, controle de jobs, pré-visualização, reimpressão, cancelamento, controle de cotas de impressão e gerenciamento de filas e prioridades.

## 2. Contexto no Sistema

Este módulo atua como o domínio central para funcionalidades relacionadas à impressão dentro do sistema web, integrando-se com serviços de usuários, controle de cotas, manipulação de arquivos e serviços de PDF. Ele provê rotas HTTP para interação via API REST, além de implementar regras de negócio específicas para o controle e autorização das impressões.

## 3. Objetivo

Permitir que usuários autorizados possam enviar documentos para impressão, controlar o histórico e o estado dos jobs, validar cotas de impressão, garantir que apenas usuários com permissão possam acessar ou manipular jobs, e fornecer funcionalidades de pré-visualização e reimpressão de documentos já impressos.

## 4. Funcionamento Geral

1. O usuário envia um arquivo para impressão via endpoint `/imprimir`, onde o arquivo é validado, convertido para PDF se necessário, e um job de impressão é criado.
2. O sistema valida a cota do usuário e as opções de impressão (cópias, páginas por folha, duplex, orientação, intervalo de páginas, tags).
3. O job é armazenado no spool (diretório configurado) e registrado no banco de dados.
4. O usuário pode visualizar o status da fila de impressão, seus próprios jobs e a cota disponível.
5. É possível pré-visualizar arquivos enviados para impressão e reimprimir jobs concluídos.
6. Jobs podem ser cancelados ou ter sua prioridade alterada por usuários com permissão.
7. O módulo também suporta impressão de anexos PDF diretamente, com validação e controle similares.
8. Políticas específicas garantem a validação dos parâmetros, controle de acesso, e regras para reutilização de jobs.

## 8. Regras de Negócio

- Apenas usuários donos do job ou gestores podem acessar ou manipular jobs.
- Jobs só podem ser reutilizados para preview ou reimpressão se estiverem com status "CONCLUIDO" ou "FINALIZADO".
- Intervalos de páginas para impressão devem ser válidos e dentro do total de páginas do documento.
- O nome do arquivo é sanitizado para evitar caracteres inválidos.
- Impressão só é permitida se a impressora estiver disponível (ex: não estiver sem papel).
- O usuário deve ter cota suficiente para a quantidade de páginas solicitadas, salvo se possuir cota ilimitada.
- É obrigatório selecionar ao menos uma tag para identificar a impressão.
- Cancelamento de jobs só é possível se o job ainda não estiver em impressão ou finalizado.
- A prioridade de jobs pode ser alterada apenas por gestores.
- Arquivos vinculados a jobs devem estar dentro do diretório spool configurado para serem reutilizados.
- Formatos aceitos para upload são PDF, DOCX, DOC, PNG, JPG ou JPEG (configurável).
- Impressões podem ser feitas em modo simplex ou duplex, com opções de orientação e páginas por folha.

## 10. Fluxo Resumido

1. Usuário envia arquivo para impressão via endpoint `/imprimir`.
2. Arquivo é armazenado no spool e convertido para PDF se necessário.
3. Parâmetros de impressão são validados (cópias, páginas, orientação, tags).
4. Cota do usuário é validada e consumida, salvo se ilimitada.
5. Job de impressão é criado e registrado no banco.
6. Usuário pode consultar status, fila, seus jobs e cota via endpoints.
7. Jobs concluídos podem ser reutilizados para preview ou reimpressão.
8. Jobs podem ser cancelados ou ter prioridade alterada por usuários autorizados.
9. Impressão direta de anexos PDF é suportada via função específica.
10. Arquivos temporários são removidos após uso ou em caso de falha.

## 11. Pontos de Atenção

- A segurança do acesso aos arquivos PDF depende da validação do caminho estar dentro do spool configurado.
- A manipulação de arquivos no sistema de arquivos pode gerar exceções que são tratadas, mas podem impactar a disponibilidade.
- A validação de tags é rígida e pode causar erros se forem enviadas tags não previstas.
- O sistema depende de serviços externos para conversão de arquivos e contagem de páginas PDF, que podem falhar.
- O cancelamento de jobs pode não ser possível se o job já estiver em impressão, o que deve ser comunicado claramente.
- A configuração do spool e impressora padrão pode vir de variáveis de ambiente ou configuração legada, o que pode causar inconsistências se não estiver bem gerenciado.
- Logs são gerados para falhas críticas, mas não há detalhamento de monitoramento ou alertas automáticos.

## 12. Sugestões de Melhoria

- Implementar cache ou otimização para leitura de arquivos PDF para preview e reimpressão.
- Adicionar suporte para mais formatos de arquivo ou melhorar a conversão para PDF.
- Melhorar mensagens de erro para o usuário final, especialmente em casos de falha de cota ou arquivo inválido.
- Implementar monitoramento e alertas para falhas frequentes na conversão ou manipulação de arquivos.
- Adicionar testes automatizados para os fluxos principais de criação, cancelamento e reimpressão.
- Documentar melhor as dependências externas e suas versões para garantir compatibilidade.
- Avaliar a possibilidade de usar armazenamento em nuvem para spool em ambientes distribuídos.
- Melhorar a interface para seleção e gerenciamento de tags, talvez com autocomplete ou validação no frontend.
